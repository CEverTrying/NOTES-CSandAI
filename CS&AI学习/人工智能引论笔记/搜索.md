一个搜索问题由以下要素组成：

- **状态空间** - 在给定世界中所有可能状态的集合  
- 每个状态下可用的**动作**集合  
- **转移**模型 - 在当前状态下执行特定动作时，输出下一个状态  
- 动作**代价**- 应用动作从一个状态转移到另一个状态时产生的代价  
- **初始状态**- 智能体最初所处的状态  
- **目标测试** - 以状态为输入，判断其是否为目标状态的函数

**状态空间（state space）**的大小：直接将不同状态的种类相乘

**状态空间图（state space graph）**：由不同的状态和箭头连接，状态不重复出现  
**搜索树**：状态可重复出现，由从上到下的树状结构表示

## 无信息搜索(uninformed search)

这种搜索的特点是**只参考已经走过的状态，对还未走到的状态不进行参考/无法参考**  
搜索的伪代码是

```
function TREE-SEARCH(problem, frontier) return a solution or failure
    frontier ← INSERT(MAKE-NODE(INITIAL-STATE[problem]), frontier)
    while not IS-EMPTY(frontier) do
        node ← POP(frontier)
        if problem.IS-GOAL(node.STATE) then return node
        for each child-node in EXPAND(problem, node) do
            add child-node to frontier
    return failure
```

其中EXPAND函数的伪代码为

```
function EXPAND(problem, node) yields nodes
    s ← node.STATE
    for each action in problem.ACTIONS(s) do
        s' ← problem.RESULT(s, action)
        yield NODE(STATE=s', PARENT=node, ACTION=action)
```

## 有信息的搜索(informed search)

 **启发（Heuristics)** 一般来说就是为之后的搜索提供一些参考/引导，一般是会告诉从当前目标到实现目标状态的**剩余代价的下界（实际代价必定大于等于这个下界）**，不能保证这个启发是准确的

| 名称                              | 贪心搜索/算法（？） | A*搜索               |
| --------------------------------- | ------------------- | -------------------- |
| 完备&最优                         | 都否                | 都是(如果Admissible) |
| 前沿表示(frontier representation) | 只看前              | 看前也看后           |

**Admissibility**(可采性?)是启发的重要评判标准,判断这个启发能否使搜索找到最优路径,要求这个启发函数的值**非负且不大于真实值**&**到目标状态时启发函数的值为0**


为了**防止搜索陷入死循环或者是浪费大量算力到已经探索的状态上**,可以记录下来已经探索过的状态，避免重复探索,这种情况下tree search就成为了graph search。
- 这里有一个问题是在Astar search中启发的值会影响最优路径的选择(如A的评估的代价过大导致A到C的搜索出现在B到C之后，但是从start经过A到C更优，如果不加改进就会选择次优的路径，即经过B的)**于是要在排除重复出现的状态之前进行比较，留下来更优的路径进行探索**
- 可以证明在UCS中不存在这个问题（~~我自己证明的不保对~~）

这里是伪代码
- 注意这里先检测是否达到了目标状态，再判断是否加入reached并且expand，可能是以防万一开始状态就是目标状态吧（）
- 无需担心第一个找到目标的路径次优，因为admissible

```
function A*-GRAPH-SEARCH(problem, frontier) return a solution or failure
    reached ← an empty dict mapping nodes to the cost to each one
    frontier ← INSERT((MAKE-NODE(INITIAL-STATE[problem]),0), frontier)
    while not IS-EMPTY(frontier) do
        node, node.CostToNode ← POP(frontier)
        if problem.IS-GOAL(node.STATE) then return node
        if node.STATE is not in reached or reached[node.STATE] > node.CostToNode then
            reached[node.STATE] = node.CostToNode
            for each child-node in EXPAND(problem, node) do
                frontier ← INSERT((child-node, child-node.COST + CostToNode), frontier)
    return failure
```

**Dominance**：如果a评估函数永远大于等于b，则称启发函数a dominate over 启发函数b

**一致性（consistency）**:单边的启发值<=实际动作代价，如A到C，h(A)-h(C)<=cost(A to C),**保证一条路径的f值不会减少**  ，==保证A* 在图搜索（去掉重复的state，再也不访问）上最优==
注：f指的是g+h，g是先前走过地方的代价，h是启发值

## 局部搜索（local search）

前面的搜索是为了**寻找目标状态的位置(或者说如何实现)并且找到一条实现(最优)路径** ,但是有时我们只需要**找到目标状态的实现方式（或者是最优化）**,这就需要局部搜索（强调在**从一个完整解开始**（不保证好），在**邻域**搜索迭代，寻找最优）了，局部搜索不涉及回溯，内存占用低，但是不能保证最优。

### 爬山法搜索(Hill-Climbing Search)

最简单的版本就是在当前状态的相邻状态中寻找增加目标值最多的路径,但是可能会困于局部最优解或者是平原.有一个变种是**随机爬山法----在相邻状态中随机选择**(是否一定选择更好的状态,概率是否固定都是视情况而定),虽然会增加计算时间但是更有可能脱离局部最优解.还有一个变种是**随机重启爬山算法(Random-restart hill-climbing)**,多次随机选择初始状态以增加找到全局最优解的概率(*但是这要算不少吧,真的是个好方法吗,感觉是高级一点点的暴力随机啊*)

### 模拟退火搜索(Simulated Annealing Search)

其要点是**一定程度上允许选择使目标值下降的路径，减少局部最优的可能性 **   
从邻居中随机选取一个移动，若使目标值增加则直接采用，若减少则概率选取(概率由变化大小和温度temperature决定),temperature依据某种"日程(schedule)"逐渐减少  
理论上如果温度降低得足够慢,算法会得到全局最优的概率会解近1

### 局部束搜索(Local Beam Search)

方法是随机k个初始状态,迭代(爬山),选择当前节点里面最优的k个,重复操作
要点是这**允许了多个线程(thread)相互交流信息,同时比较节省内存**  
这个算法和爬山同样容易困在平原,随机局部束搜索能缓解这个问题
